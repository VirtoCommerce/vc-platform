SECTION 01

The architecture
at a glance

How modularity works across the three backoffice host UIs — and why the contract is built once, used everywhere.

Architecture · Hosts

Three host flavours, one contract

Different UI tech, same plugin discovery rules — module authors learn one workflow, regardless of which admin app the plugin targets.

Platform Admin

Mature AngularJS 1.8 admin used in production for years.

Legacy loader

Convention: dist/app.js + dist/style.css per module.

VC-Shell

Modern Vue 3 admin with Module Federation, used by Marketplace and vertical apps.

MF loader

defineAppModule, blades, extension points.

Custom SPAs

Self-contained apps served via the platform's <apps> mechanism.

MF loader

Example: System Operations — Vue 3 SPA with its own host API.

Why one contract matters. Before the framework: every host had its own way to load plugins (AngularJS implicit bundles · stub POST /frontend-modules for VC-Shell · custom SPAs not extensible at all) and frontend modularity sat in a separate registry from backend. Now: module.manifest declares modularity for both backend and frontend, and GET /api/apps/{appId}/manifest serves all three host UIs.

Architecture · Layers

Platform · Host · Plugin

Three layers, each with one job. The platform tells the host which plugins to load; the host loads them; the plugin contributes UI.

Architecture · Lifecycle

Plugins ship inside VC Modules

A "plugin" is not a separate thing with its own marketplace. It is — quite literally — files inside a VC Module. The whole platform install / uninstall / version / dependency machinery applies for free.

One distribution unit

The VC Module is the only thing that gets shipped, installed, versioned, uninstalled. Plugins are files inside it — not a parallel artefact.

Standard install path

Same install / uninstall as any other VC Module: vc-build, vc-package, VC Module Manager UI, CI/CD pipelines. Operators learn nothing new.

Lifecycle for free

Uninstall a module → its plugins disappear from manifest. Bump module version → cache invalidates on next call. Dependency conflicts caught at install time.

Architecture · API

One endpoint to rule them all

The platform exposes a single, cacheable endpoint. Every host UI consumes the same JSON shape — only the loader strategy differs.

GET /api/apps/{appId}/manifest 
{
  "appId":   "vc-shell-marketplace",
  "version": "3.1000.0",
  "title":   "Marketplace",
  "hash":    "8DBA4F3C9E2A...",
  "plugins": [
    {
      "id":         "VirtoCommerce.MarketplaceReviews",
      "version":    "3.1001.0",
      "permission": null,
      "entry": {
        "type": "script",
        "path": "/modules/.../remoteEntry.js",
        "hash": "8DBA4F3C"
      },
      "remote": { "name": "…", "exposed": "./Module" }
    }
  ]
}

What's in it

  • Topologically sorted — plugins ship in module-dependency order.
  • Cache-busting hashes — per-file mtime + descriptor-level hash as ETag.
  • Permission metadata — host SPA evaluates plugin.permission client-side.
  • Federation coordinates — name + exposed for loadRemote().

Cacheability

If-None-Match against hash → 304 Not Modified in microseconds. Server-side cache via IPlatformMemoryCache for the process lifetime — no filesystem walk on cache-hits.

Architecture · Runtime

How a plugin gets loaded

Five steps from page load to rendered plugin. The host shell paints first; plugins flow in reactively as remotes resolve.

Architecture · Discovery

Drop a folder, ship a plugin

No new XML element, no plugin registry, no marketplace ID. The platform finds plugins by walking the dependency graph it already maintains.

Source repo (authoring)
// vc-module-system-operations/samples/
└── VirtoCommerce.SystemOperations.SampleExtension/
    ├── _module.manifest         ← <dependency id="…SystemOperations">
    ├── package.json
    ├── vite.config.mts          ← @module-federation/vite
    ├── src/
    │   ├── index.ts             ← default export install()
    │   ├── BrowserInfoCard.vue
    │   └── host-types.ts
    └── plugins/                 ← Vite outputs here
        └── system-operations/   ← appId of target host
            ├── remoteEntry.js
            ├── index.js
            └── assets/
Installed (on the platform)
// platform/modules/
└── VirtoCommerce.SystemOperations.Ext/
    ├── module.manifest
    └── plugins/
        └── system-operations/
            ├── remoteEntry.js   ← MF entry
            ├── index.js
            └── assets/

Discovery mechanism

Plugin's module.manifest declares <dependency id="HostModule">. Platform walks the topologically sorted module list and probes {moduleRoot}/plugins/{appId}/remoteEntry.js — found = plugin.

One module · multiple apps

The folder name under plugins/ is the target appId, not the plugin id. A single module can host plugins for several host apps: 

plugins/
├── system-operations/      ← for SO host
├── vc-shell-marketplace/   ← for Marketplace
└── platform/               ← for legacy admin

One manifest, backend + frontend

module.manifest is the single declaration. No second registry. Frontend-only modules just leave <assemblyFile> empty — pure JS ships like any other module.

Architecture · Security

Two layers of permission

App access is gated server-side; plugin visibility is gated client-side. Both leverage the platform's existing permission system.

Server-side: app permission

<app permission="my-app:access"> on the host module. If the user lacks the permission, the manifest endpoint returns 403 Forbidden — the user never receives the plugin list at all. 

<!-- module.manifest -->
<apps>
  <app id="vendor-portal">
    <title>Vendor Portal</title>
    <permission>vendor_portal:access</permission>
  </app>
</apps>

Client-side: plugin permission

plugin.json declares a permission. The host SPA evaluates it before loadRemote(). Why client-side? The manifest is cached once across all users — server-side filter would multiply cache entries by user count. 

// plugins/system-operations/plugin.json
{
  "permission": "system-operations:advanced"
}

The plugin itself enforces deeper authorization on the API endpoints it calls — defense in depth.

Architecture · Performance

Cache like a pro

Three layers. Each plays a different role per environment. Production fast paths · dev workflow that just works.

Environment Backend cache Cache-Control Behaviour
Production IPlatformMemoryCache · process lifetime private, must-revalidate First request — full body. Subsequent — 304 in microseconds.
Development Bypassed entirely no-store Every request — fresh build. Plugin rebuild visible on next reload, no platform restart.
After invalidate Rebuilt on next call (unchanged from env) Browser revalidates with old ETag → 200 with new body if changed.

Force-invalidate without restart

POST /api/apps/manifest/invalidate
Authorization: Bearer …
// requires platform:module:manage

→ 204 No Content

What this gets you

  • Cold start ~ms; warm requests microseconds.
  • Dev workflow: yarn build → reload, no restart.
  • Operators can hot-swap plugin bundles via single POST.
  • Browser always sees fresh URLs when files change (mtime-based hashes).
Architecture · Versioning

One mechanism, no second registry

Compatibility piggybacks on the platform's existing <dependency> graph — the same one operators already understand. No separate frontend-versioning system to learn.

Check Where declared Validated when Gates
Module dependency <dependency id version> Module install time Whether the plugin module is allowed to install. Single source of truth for both backend and frontend compatibility.
MF shared (runtime) federation({ shared: { vue: { requiredVersion } } }) loadRemote() shared negotiation Whether the plugin uses the host's Vue / Router / etc., or its own copy.

One manifest, one dependency graph. Earlier drafts had a separate <frontendModules compatibleWith> registry — now obsolete. The unified module.manifest handles backend + frontend in one place; bumping <dependency version> on the host module is the only declaration plugin authors need to make.

SECTION 02

Manifest Settings

Declarative platform settings in module.manifest — global and per-user, no programmatic registration required.

Manifest Settings · Concept

Settings as configuration, not code

A module declares its settings in XML. The platform parses, validates, and registers them. The same v2 settings API serves them — no new endpoints, no new admin UI plumbing.

Before

Every module had to register settings programmatically from IModule.Initialize(). Frontend-only modules can't. 

public void Initialize(IServiceCollection services)
{
    var registrar = services.BuildServiceProvider()
        .GetRequiredService<ISettingsRegistrar>();

    registrar.RegisterSettings(
        ModuleConstants.Settings.AllSettings,
        ModuleInfo.Id);
}

After

Drop a <settings> element into module.manifest. Platform parses + registers automatically. Works for any module — assembly or not. 

<settings>
  <setting>
    <name>VirtoCommerce.MyExt.MaxRetries</name>
    <groupName>MyExt|Reliability</groupName>
    <valueType>PositiveInteger</valueType>
    <defaultValue>3</defaultValue>
  </setting>
</settings>
Manifest Settings · Tenant types

One mechanism, two scopes

A single tenant= attribute decides whether a setting is global (shared across all admins) or per-user (theme, layout density, feature flags).

Manifest Settings · UserProfile

Per-user settings, across devices

A user changes their theme on one browser → it follows them to another. The platform stores per-user values under the authenticated caller's profile, surfaced via dedicated /me/* endpoints.

Manifest 
<setting tenant="UserProfile">
  <name>VC.SystemOperations.DefaultTheme</name>
  <groupName>System Operations|UI</groupName>
  <displayName>Theme preference</displayName>
  <valueType>ShortText</valueType>
  <defaultValue>system</defaultValue>
  <allowedValues>
    <value>system</value>
    <value>light</value>
    <value>dark</value>
  </allowedValues>
</setting>

Self-service /me/* endpoints

Three endpoints resolve {userId} from the auth token — frontend never sends it.

GET /v2/me/schema?moduleId=…
GET /v2/me/values?modifiedOnly=…
POST /v2/me/values

Auth-only. Your own profile is yours — no platform:setting:* permission needed for /me/*. Cross-user access uses the existing tenant endpoints.

Manifest Settings · Frontend

One composable, two scopes

The Vue 3 storefront and System Operations ship a tiny reactive composable. Same API regardless of scope — scope: 'UserProfile' swaps the underlying endpoint family.

// Vue 3 storefront / custom SPA
import { useModuleSettings } from './composables/useModuleSettings';

// global tier
const globalSettings = useModuleSettings('VC.SystemOperations');

// per-user tier
const userSettings = useModuleSettings('VC.SystemOperations', { scope: 'UserProfile' });

// usage
await globalSettings.load();
const timeout = globalSettings.get<number>('VC.SystemOperations.RestartTimeoutSeconds', 120);

await userSettings.load();
await userSettings.save({ 'VC.SystemOperations.DefaultTheme': 'dark' });
// → POST /api/platform/settings/v2/me/values  ·  persists to the caller's profile

One round-trip

Single GET to /values?moduleId=…&modifiedOnly=false — values + defaults inline. No separate /schema fetch.

Reactive

Returns a Ref; UI re-renders on save() optimistic merge. No second round-trip.

Best-effort load

401/403/network → fall back to caller fallback. Page renders with defaults first, refines after load.

SECTION 03

What this unlocks

For module authors, platform operators, and product owners — what changes once the contract is in place.

Value · Outcomes

Why it matters

Architecture decisions usually have invisible payoff. Here's what becomes possible — or trivially possible — with this framework.

For module authors

  • One toolchain (@module-federation/vite) for any host.
  • Drop a folder, ship a plugin — no plugin.json in 80% of cases.
  • Settings declared in XML — no code for frontend-only modules.
  • NSwag-generated TS clients for the API.

For operators

  • One config surface (module.manifest) for every kind of extension.
  • One API to introspect plugins (curl-friendly).
  • Hot-swap plugin bundles via single POST — no restart.
  • Permission-gated apps prevent accidental access.

For the product

  • Independent release cycles for vertical apps + plugins.
  • Backwards compatible — every legacy AngularJS module works unchanged.
  • Marketplace-ready foundation for third-party plugins.
  • Per-user preferences without writing storage code.

Bottom line. The framework removes friction at every layer — author, operator, product. New plugins reach production faster, with less plumbing, on a stable contract that scales from a single team to a marketplace ecosystem.

Value · Roadmap

What's next

The current contract is v1. Several capabilities are already on the roadmap or under consideration.

Near-term

  • Public manifest endpoint for anonymous storefront-style apps — CDN-friendly cache headers, dedicated path.
  • Runtime resilience baseline — bundled retry + errorLoadRemote in @vc-shell/mf-host by default.
  • DTS plugin migration — replace inline host-types with @mf-types.zip auto-generation.
  • vc-frontend extensibility — Phase 1 from the storefront roadmap (extension points without MF).

Longer-term

  • Hot reload end-to-end — in-place plugin swap with state preservation.
  • Cross-framework plugins — React + Vue interop via MF Bridge.
  • Custom tenant types in declarative settings — tenant="Store", "Organization".
  • Marketplace — third-party plugins with security review, sandboxing, signed manifests.

Modular by design

One contract. Three host UIs. Zero-config discovery. Manifest-declared settings. A foundation that scales from one team to a plugin marketplace.
