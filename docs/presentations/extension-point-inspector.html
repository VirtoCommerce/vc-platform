Adding a widget to a blade — before & after

A concrete onboarding task. The mechanics haven't changed; what changed is how long it takes and how often it ships broken on the first try.

!Before

Hunt-and-peck the docs

  1. 1Open the extensibility doc page; find the section on widgets
  2. 2grep -r "registerWidget" modules/ — get a wall of results
  3. 3Guess the container name: orderDetail? customerOrderDetail? customerOrderDetailWidgets?
  4. 4Write the snippet by hand; copy template path conventions from another module
  5. 5Reload. Widget doesn't appear. Wrong group. Repeat from step 3.
  6. 6Eventually ask a teammate
Time to first working build 30 – 60 min
→After

Highlight, copy, paste

  1. 1Open the page where you want to add the widget
  2. 2DevTools console → vcExt.show()
  3. 3The widget container is outlined green and labelled
    [widget] :: customerOrderDetailWidgets (12 widgets)
  4. 4Click Copy snippet on that label
  5. 5Paste into your module's .run block; replace controller + template paths
  6. 6Reload — widget appears. Right name, every time.
Time to first working build < 1 min
Overview

Executive summary

Why this exists, what it does, and what you get.

!Problem

The hidden contracts

  • Extension-point names are invisible in the UI — widget container groups, toolbar controllers, meta-form keys
  • Docs list them in the abstract; mapping a doc entry to this screen is manual
  • Wrong identifier = silent failure: registration succeeds but UI never updates
  • Onboarding new module authors burns hours on copy-paste plumbing
→Solution

Highlight everything, copy anything

  • One console command — vcExt.show() — outlines every extension point
  • Each highlight shows the actual id the developer needs
  • One-click Copy snippet writes a parameterised registration call to the clipboard
  • Always loaded, always available — invisible until invoked
★Features

What you get

  • 5 extension-point types: menu, blade, toolbar, widget, meta-form
  • Bare service-call snippets (paste into your module's .run)
  • Auto-refresh as blades open and close
  • Toast confirms each copy; vcExt.list() for full inventory
SECTION 01

What it is —
and what it isn't

A read-only inspection tool. No registry mutations, no permission gating, no UI affordance until you ask for it.

What it is · Console API

Six commands, one global

On bundle load the inspector exposes window.vcExt with a tiny, discoverable surface. No setup, no permission, no menu item.

CommandWhat it does
vcExt.help()Print usage to console
vcExt.show()Overlay every extension point on the current page (idempotent)
vcExt.hide()Tear down overlay; detach observer + listeners
vcExt.toggle()Show / hide
vcExt.list()console.table of every registered item across all five services
vcExt.copy(
  type, id?)		Copy a registration snippet for the given type
('menu' | 'blade' | 'toolbar' | 'widget' | 'metaform')

On bundle load

// the only output until you call vcExt.show()
[Virto] Extension-point inspector loaded.
        Type vcExt.help() to begin.

Production safety

Bundled into dist/app.js via the existing webpack glob — no _Layout.cshtml change, no new endpoint, no new permission. Bundle delta ≈ 8 KB. Vanilla IIFE; injector calls are deferred until show(), so it never interferes with Angular bootstrap.

What it is · Coverage

Five extension points, colour-coded

The five most-extended points in Platform Manager. Each maps to a stable DOM marker AND an AngularJS service registry — so the overlay can show the count of currently-registered items.

Type DOM selector Service / registry What you extend
Main menu nav.nav-bar mainMenuService.menuItems Top-level navigation entries
Blade [ng-model="blade"] bladeNavigationService.blades Detail / edit panels
Blade toolbar .blade-toolbar toolbarService
.toolbarCommandsMap[ctrl]		 Buttons in a blade's header
Widget container [gridster-opts] widgetService.widgetsMap[group] Tiles inside a blade or dashboard
Meta form [registered-inputs] metaFormsService.getMetaFields(name) Dynamic form fields on a blade

Why these five. They are the canonical list at docs.virtocommerce.org/.../key-extensibility-points/. Push-notification templates, setup-wizard steps and breadcrumb history are exposed via vcExt.list() only — they have no stable on-screen DOM host to highlight.

SECTION 03

What ends up in
your clipboard

Bare service-call snippets — no module bootstrap wrapper. Paste into your existing module's .run block and you're done.

Snippets · Navigation

Menu · Blade · Toolbar

The three navigation primitives. Each snippet is parameterised to the highlighted host — the toolbar copy carries its host blade's controller name; the blade copy carries its id.

[menu]
mainMenuService.addMenuItem({
  path:     '<group>/<item>',
  icon:     'fa fa-cube',
  title:    'platform.menu.<key>',
  priority: 100,
  action:   function () {
    $state.go('<state-name>');
  },
  permission: '<permission>'
});
[blade] :: orders
var blade = {
  id:         'orders',
  title:      'platform.blades.<key>',
  controller: '<your.controller>',
  template:
    '$(YourModule)/Scripts/blades/orders.tpl.html'
};
bladeNavigationService.showBlade(
  blade, parentBlade
);
[toolbar] :: orderModule…operationDetailCtrl
toolbarService.register({
  name:  'platform.commands.<k>',
  icon:  'fa fa-cog',
  index: 100,
  executeMethod:
    function (blade) { /* … */ },
  canExecuteMethod:
    function (blade) { return true; }
}, 'virtoCommerce.orderModule.…');

Format note. Bare service call. No angular.module(…).run([…]) wrapper, no wiring boilerplate. Drop it into your module's existing .run block alongside other registrations.

Snippets · Content

Widget · Meta-form

The two content extension points. Widget snippets carry the exact container group; meta-form snippets carry the resolved form name (or a placeholder if heuristic resolution can't pin it down).

[widget] :: customerOrderDetailWidgets
widgetService.registerWidget({
  controller: '<yourModule>.<yourWidgetController>',
  template:   '$(YourModule)/Scripts/widgets/<widget>.tpl.html'
}, 'customerOrderDetailWidgets');
The exact group name is read from the widget container's group attribute. No more guessing between customerOrderDetailWidgets, orderDetail, and customerOrderDetail.
[metaform] :: accountDetails
metaFormsService.registerMetaFields('accountDetails', [{
  name:      '<fieldName>',
  title:     'platform.blades.<form>.labels.<f>',
  valueType: 'ShortText',
  priority:  100
}]);
The form name is recovered by reverse-lookup: comparing the directive's bound array reference against metaFormsService.getMetaFields(name) for known seed names. Custom forms fall back to a '<formName>' placeholder.
SECTION 04

Where the same idea
goes next

The Platform Manager is the first host. The same console-gated overlay pattern applies cleanly to vc-frontend (Vue 3), VC-Shell, and any custom SPA built on the modularity framework.

What's next · Roadmap

Same pattern, different host

Port the inspector to vc-frontend (Vue 3 storefront) and VC-Shell (Module Federation admin). The UX contract is identical — only the detection layer changes.

vc-frontend (Vue 3)

Detect Vue named slots, Pinia plugin registries, route slot outlets. Add data-vc-slot attributes behind import.meta.env.DEV; the rest of the inspector runs unchanged.

Storefront

VC-Shell (Vue 3 + MF)

Detect MF remote-loaded blades and extension points. Inspect the defineAppModule registry. Snippet output uses VC-Shell's plugin install() idiom.

Admin (modern)

Custom SPAs

System Operations and other vertical SPAs. The same console API (vcExt.show()) gives a uniform experience across every Virto host UI.

Vertical apps

A self-contained prompt for replicating the inspector in a new host UI is checked into docs/prompts/. The same console API (vcExt.show / hide / list / copy) and colour scheme apply across every Virto host — module authors learn the tool once.

Highlight, copy, ship

One console command, five extension points outlined in colour, snippets a click away. The kind of tool you should have had on day one.
